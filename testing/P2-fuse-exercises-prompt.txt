====================================================================
PROMPT P2 — FUSION DES DOUBLONS DU CATALOGUE D'EXERCICES (6way)
À coller dans Lovable — exécuter UNIQUEMENT APRÈS P1 appliqué
====================================================================

Crée une nouvelle Supabase edge function `fuse-duplicate-exercises` (TypeScript/Deno).

Cette fonction fusionne les exercices par défaut en double dans `public.exercises`.
Elle repointe les FK, purge les orphelins, supprime les perdants, et vérifie l'intégrité finale.

**Prérequis :** la fonction P1 `normalize-exercises-catalog` doit avoir été exécutée avec dry_run=false avant d'appeler celle-ci.

---

## Pattern de base

Même structure que `create-test-accounts/index.ts` :
- Client admin `SUPABASE_SERVICE_ROLE_KEY`
- corsHeaders `"*"`, gestion OPTIONS
- JSON body en entrée

---

## Interface d'appel

```ts
{
  dry_run: boolean;    // défaut true
  confirm?: string;    // "APPLIQUER_FUSIONS" pour autoriser les écritures
}
```

---

## Périmètre FK — 4 tables à repointer

Pour chaque fusion (perdant → survivant), repointer dans cet ordre :
1. `session_exercises` (colonne `exercise_id`)
2. `exercise_favorites` (colonne `exercise_id`) — contrainte UNIQUE probable → `ON CONFLICT DO NOTHING`
3. `skipped_exercises` (colonne `exercise_id`) — contrainte UNIQUE probable → `ON CONFLICT DO NOTHING`
4. `exercise_video_requests` (colonne `exercise_id`)

Après le repointage avec ON CONFLICT DO NOTHING, supprimer les lignes du perdant qui n'ont pas pu être repointées (orphelines) :
```sql
DELETE FROM exercise_favorites WHERE exercise_id = <id_perdant>;
DELETE FROM skipped_exercises WHERE exercise_id = <id_perdant>;
```

Seulement ensuite : `DELETE FROM exercises WHERE id = <id_perdant>`.

**Vérification préalable au premier appel :** avant tout dry-run, exécuter cette requête et inclure son résultat dans la réponse pour confirmer qu'aucune autre table ne référence exercises.id :
```sql
SELECT tc.table_name, kcu.column_name
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
  ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.referential_constraints rc
  ON tc.constraint_name = rc.constraint_name
JOIN information_schema.table_constraints tc2
  ON rc.unique_constraint_name = tc2.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND tc2.table_name = 'exercises';
```

### Groupe B — ID à résoudre par nom (relecture visuelle)

Pour chaque entrée, résoudre les id via requête SQL avant d'afficher le dry-run :

```sql
SELECT id, name, name_en, muscle_group, equipment, type,
  (SELECT COUNT(*) FROM session_exercises WHERE exercise_id = e.id) +
  (SELECT COUNT(*) FROM exercise_favorites WHERE exercise_id = e.id) +
  (SELECT COUNT(*) FROM skipped_exercises WHERE exercise_id = e.id) +
  (SELECT COUNT(*) FROM exercise_video_requests WHERE exercise_id = e.id) AS total_refs
FROM exercises e
WHERE is_default = true AND created_by IS NULL
  AND (
    lower(trim(coalesce(name_en, ''))) IN (<liste_des_noms_en_lowercase>) OR
    lower(trim(name)) IN (<liste_des_noms_fr_lowercase>)
  );
```

## Corrections de type sur exos distincts (Phase 3)

Après les fusions, appliquer ces corrections sur des exos distincts conservés :

```sql
-- Hip thrust machine : isolation → compound
UPDATE exercises
SET type = 'compound'
WHERE is_default = true AND created_by IS NULL
  AND (lower(name) LIKE '%hip thrust%machine%' OR lower(name_en) LIKE '%hip thrust%machine%')
  AND type = 'isolation';

-- Face pull élastique : compound → isolation
UPDATE exercises
SET type = 'isolation'
WHERE is_default = true AND created_by IS NULL
  AND (lower(name) LIKE '%face pull%élastique%' OR lower(name_en) LIKE '%face pull%elastic%' OR lower(name_en) LIKE '%face pull%band%')
  AND type = 'compound';
```

## Comportement dry_run=true

**Étape 1 — Vérification FK :** exécuter la requête de détection des FK sur `exercises` (voir section "Périmètre FK") et inclure le résultat.

**Étape 2 — Résolution des id :** pour chaque fusion du Groupe B, résoudre les id par SELECT et construire la liste `{ cluster, survivant_id, survivant_name, perdants: [{id, name, total_refs}] }`.

Pour les clusters marqués `dynamic_survivant: true` (treadmill), choisir comme survivant l'entrée avec le plus de refs total.

**Étape 3 — Rapport :** retourner :

```json
{
  "mode": "DRY_RUN — aucune écriture",
  "fk_autres_tables": [...],  // résultat de la vérification FK
  "fusions": [
    {
      "cluster": "crunch",
      "survivant": { "id": "...", "name": "Crunch", "total_refs": 42 },
      "perdants": [
        { "id": "...", "name": "Crunch classique", "total_refs": 0,
          "fk_a_repointer": { "session_exercises": 0, "exercise_favorites": 0, "skipped_exercises": 0, "exercise_video_requests": 0 }
        }
      ],
      "type_final": "isolation",
      "muscle_group_final": "Abdominaux",
      "statut": "CONFIRMÉ"
    },
    {
      "cluster": "squat_triplon",
      "flag": "ARBITRAGE_4_A_VALIDER",
      "survivant": { ... },
      "perdants": [...],
      "statut": "⚠️ À VALIDER — inclus par défaut, retirer du body si non souhaité"
    }
    // ... tous les clusters
  ],
  "total_exercices_a_supprimer": N,
  "total_refs_a_repointer": N,
  "next_step": "Relancer avec { dry_run: false, confirm: 'APPLIQUER_FUSIONS' } pour appliquer."
}
```

Les clusters marqués `flag` sont inclus dans le dry-run avec un warning — ils sont appliqués par défaut sauf si l'appelant les exclut via un paramètre `skip_clusters: string[]`.

## Comportement dry_run=false

Guard :
```ts
if (!body.dry_run && body.confirm !== "APPLIQUER_FUSIONS") {
  return new Response(JSON.stringify({ error: "confirm manquant ou invalide" }), { status: 400 });
}
```

Pour chaque fusion (dans l'ordre : Groupe A d'abord, Groupe B ensuite) :

```ts
// Pour chaque perdant d'une fusion :
// 1. Repointer session_exercises
await admin.rpc('exec_sql', `
  UPDATE session_exercises SET exercise_id = '${survivant_id}'
  WHERE exercise_id = '${perdant_id}'
`);

// 2. Repointer exercise_favorites (ON CONFLICT DO NOTHING)
await admin.rpc('exec_sql', `
  UPDATE exercise_favorites SET exercise_id = '${survivant_id}'
  WHERE exercise_id = '${perdant_id}'
  ON CONFLICT (user_id, exercise_id) DO NOTHING
`);
// Purger les orphelins restants
await admin.rpc('exec_sql', `DELETE FROM exercise_favorites WHERE exercise_id = '${perdant_id}'`);

// 3. Repointer skipped_exercises (ON CONFLICT DO NOTHING)
await admin.rpc('exec_sql', `
  UPDATE skipped_exercises SET exercise_id = '${survivant_id}'
  WHERE exercise_id = '${perdant_id}'
  ON CONFLICT (user_id, exercise_id) DO NOTHING
`);
await admin.rpc('exec_sql', `DELETE FROM skipped_exercises WHERE exercise_id = '${perdant_id}'`);

// 4. Repointer exercise_video_requests
await admin.rpc('exec_sql', `
  UPDATE exercise_video_requests SET exercise_id = '${survivant_id}'
  WHERE exercise_id = '${perdant_id}'
`);

// 5. Appliquer type_final et muscle_group_final sur le survivant si différent
await admin.rpc('exec_sql', `
  UPDATE exercises
  SET type = '${type_final}', muscle_group = '${muscle_group_final}'
  WHERE id = '${survivant_id}'
`);

// 6. Renommer si cluster leg_curl_generique
// (renommer survivant avant la fusion)

// 7. DELETE le perdant
await admin.rpc('exec_sql', `DELETE FROM exercises WHERE id = '${perdant_id}'`);
```

**Idempotence :** avant chaque DELETE, vérifier que l'id existe encore (`SELECT id FROM exercises WHERE id = perdant_id`) — si absent (déjà supprimé lors d'un run précédent), passer au suivant sans erreur.

## Sanity checks post-fusions (inclure dans la réponse)

```sql
-- 1. Aucun doublon exact résiduel
SELECT lower(trim(coalesce(name_en, name))) as key, COUNT(*) as nb
FROM exercises
WHERE is_default = true AND created_by IS NULL
GROUP BY key HAVING COUNT(*) > 1;
-- → doit retourner 0 lignes

-- 2. Aucune FK orpheline
SELECT COUNT(*) FROM session_exercises WHERE exercise_id NOT IN (SELECT id FROM exercises);
SELECT COUNT(*) FROM exercise_favorites WHERE exercise_id NOT IN (SELECT id FROM exercises);
SELECT COUNT(*) FROM skipped_exercises WHERE exercise_id NOT IN (SELECT id FROM exercises);
SELECT COUNT(*) FROM exercise_video_requests WHERE exercise_id NOT IN (SELECT id FROM exercises);
-- → tous doivent retourner 0

-- 3. Comptage final
SELECT COUNT(*) as total_default_exercises
FROM exercises WHERE is_default = true AND created_by IS NULL;
-- → attendu ≈ 177 (200 − ~23 supprimés)

-- 4. Aucune valeur EN résiduelle
SELECT muscle_group, COUNT(*) FROM exercises
WHERE is_default = true AND created_by IS NULL
  AND muscle_group IN ('glutes','legs','abs','chest','shoulders','arms','back')
GROUP BY muscle_group;
-- → doit retourner 0 lignes (P1 doit avoir été appliqué)
```

Retourner le résultat de chaque check dans `"sanity_checks"` de la réponse.
