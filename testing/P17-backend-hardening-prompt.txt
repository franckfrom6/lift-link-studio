### Fix 1 — `supabase/functions/youtube-search/index.ts` : exiger l'authentification

La fonction est appelable par n'importe qui (CORS *, pas de check JWT) → le quota
YouTube API peut être épuisé par un tiers. Ajoute en tête de handler (après le
traitement OPTIONS) :

```typescript
const authHeader = req.headers.get("authorization");
if (!authHeader?.startsWith("Bearer ")) {
  return new Response(JSON.stringify({ error: "Unauthorized" }), {
    status: 401, headers: { ...corsHeaders, "Content-Type": "application/json" },
  });
}
const supabaseAuth = createClient(
  Deno.env.get("SUPABASE_URL")!,
  Deno.env.get("SUPABASE_ANON_KEY")!,
  { global: { headers: { Authorization: authHeader } } }
);
const { data: { user }, error: authError } = await supabaseAuth.auth.getUser();
if (authError || !user) {
  return new Response(JSON.stringify({ error: "Unauthorized" }), {
    status: 401, headers: { ...corsHeaders, "Content-Type": "application/json" },
  });
}
```

(Importer `createClient` depuis esm.sh comme dans les autres fonctions. Vérifie que
le client front passe déjà le header Authorization via `supabase.functions.invoke` —
c'est automatique.)

### Fix 2 — `supabase/functions/send-push/index.ts` : même check d'auth

La fonction est aujourd'hui un placeholder (elle logge seulement), mais elle est
ouverte à tous. Ajoute le MÊME bloc d'authentification que Fix 1, PLUS une
vérification d'autorisation sur `studentId` :

```typescript
// Le caller doit être le student lui-même OU son coach actif
const serviceClient = createClient(
  Deno.env.get("SUPABASE_URL")!,
  Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!
);
const isSelf = user.id === studentId;
let isCoach = false;
if (!isSelf) {
  const { data: rel } = await serviceClient
    .from("coach_students")
    .select("id")
    .eq("coach_id", user.id)
    .eq("student_id", studentId)
    .eq("status", "active")
    .maybeSingle();
  isCoach = !!rel;
}
if (!isSelf && !isCoach) {
  return new Response(JSON.stringify({ error: "Forbidden" }), {
    status: 403, headers: { ...corsHeaders, "Content-Type": "application/json" },
  });
}
```

### Fix 6 — Migration RLS : WITH CHECK manquant

La policy UPDATE `athlete_can_soft_delete_sessions`
(migration `20260314163920_*`) a un `USING` mais pas de `WITH CHECK`. Crée une
nouvelle migration :

```sql
-- Ajoute le WITH CHECK manquant à la policy de soft-delete athlète.
-- (DROP + CREATE car ALTER POLICY ne permet pas d'ajouter WITH CHECK seul partout)
DROP POLICY IF EXISTS "athlete_can_soft_delete_sessions" ON public.sessions;
CREATE POLICY "athlete_can_soft_delete_sessions"
  ON public.sessions
  FOR UPDATE
  TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM program_weeks pw
      JOIN programs p ON p.id = pw.program_id
      WHERE pw.id = sessions.week_id AND p.student_id = auth.uid()
    )
  )
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM program_weeks pw
      JOIN programs p ON p.id = pw.program_id
      WHERE pw.id = sessions.week_id AND p.student_id = auth.uid()
    )
  );
```

⚠️ IMPORTANT : avant d'écrire la migration, lis la policy EXISTANTE dans
`supabase/migrations/20260314163920_9ee1d421-f9ce-44ac-8518-628f28723b25.sql` et
recopie EXACTEMENT sa condition USING actuelle dans les deux clauses — ne pas
inventer une condition différente.
