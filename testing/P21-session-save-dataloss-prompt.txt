## P21 — Pertes de données silencieuses à l'enregistrement des séances run & hybride

### Contexte
Revue de code : deux écritures Supabase à la fin d'une séance ne vérifient pas leur
résultat. En cas d'échec (réseau, RLS, contrainte), l'utilisateur voit malgré tout
« Séance enregistrée ! » alors que les données sont perdues. Le coach ne voit ensuite
aucun log. Ce sont des bugs de **perte de données**, prioritaires.

(Note : le bug de réordonnancement coach de P13 est déjà corrigé sur main — ne pas y retoucher.)

---

### Fichier : `src/pages/student/HybridLiveSession.tsx`

#### Fix 1 — `onSave` : vérifier l'insert des `hybrid_block_executions` + ne plus avaler l'erreur

Localise le callback `onSave` (vers la ligne ~338). L'insert `completed_sessions` est
correctement vérifié (`csErr`), mais l'insert `hybrid_block_executions` (vers la ligne ~358)
ne l'est PAS, et le `catch {}` final n'a ni variable ni log. Résultat : si les blocs ne
s'insèrent pas, la séance est marquée terminée mais le détail (cardio/muscu) est perdu,
sans aucune trace.

Remplace le bloc de l'insert des blocs :

```typescript
          if (blockExecutions.length > 0) {
            await supabase.from("hybrid_block_executions").insert(
              blockExecutions.map((be) => ({
                completed_session_id: cs.id,
                block_id: be.block_id,
                status: be.status,
                skip_reason: be.skip_reason || null,
                log_data: {
                  cardio_log: be.cardio_log,
                  strength_log: be.strength_log,
                  mixed_log: be.mixed_log,
                } as any,
              })),
            );
          }

          toast.success("Séance enregistrée !");
          navigate("/student");
        } catch {
          toast.error("Erreur lors de l'enregistrement");
        }
```

par :

```typescript
          if (blockExecutions.length > 0) {
            const { error: beErr } = await supabase
              .from("hybrid_block_executions")
              .insert(
                blockExecutions.map((be) => ({
                  completed_session_id: cs.id,
                  block_id: be.block_id,
                  status: be.status,
                  skip_reason: be.skip_reason || null,
                  log_data: {
                    cardio_log: be.cardio_log,
                    strength_log: be.strength_log,
                    mixed_log: be.mixed_log,
                  } as any,
                })),
              );
            if (beErr) throw beErr;
          }

          toast.success("Séance enregistrée !");
          navigate("/student");
        } catch (e) {
          console.error("[HybridLiveSession] save failed:", e);
          toast.error("Erreur lors de l'enregistrement");
        }
```

(L'insert `completed_sessions` au-dessus reste inchangé — il est déjà vérifié.)

### Fichier : `src/pages/student/RunningLiveSession.tsx`

#### Fix 2 — `handleSaveFinal` : vérifier l'insert des `completed_sessions`

Localise `handleSaveFinal` (vers la ligne ~195). Le premier insert (vers la ligne ~199)
n'est pas vérifié : s'il échoue mais que l'insert `external_sessions` qui suit réussit, on
se retrouve avec une activité externe enregistrée sans séance complétée correspondante.

Remplace :

```typescript
      await supabase.from("completed_sessions").insert([{
        student_id: studentId,
        session_id: sessionId,
        started_at: startTime.toISOString(),
        completed_at: new Date().toISOString(),
        duration: sessionElapsed,
      }]);
```

par :

```typescript
      const { error: csErr } = await supabase.from("completed_sessions").insert([{
        student_id: studentId,
        session_id: sessionId,
        started_at: startTime.toISOString(),
        completed_at: new Date().toISOString(),
        duration: sessionElapsed,
      }]);
      if (csErr) throw csErr;
```

(Le `try/catch` existant de `handleSaveFinal` attrape déjà l'exception et affiche le toast
d'erreur ; vérifie juste que le `catch` logge l'erreur en console — ajoute
`console.error("[RunningLiveSession] save failed:", e)` s'il ne le fait pas déjà.)

### Comportement attendu
- Séance hybride : si les blocs ne s'enregistrent pas, l'utilisateur voit l'erreur (pas un
  faux succès), et l'échec est tracé en console
- Séance run : un échec d'enregistrement de la séance complétée n'aboutit plus à une
  activité externe orpheline ; l'utilisateur voit l'erreur

### Contraintes
- Ne pas modifier la logique d'exécution des séances, uniquement la gestion d'erreur à
  l'enregistrement
- Ne pas toucher au réordonnancement coach (déjà corrigé)
- Tous les logs préfixés par le nom du composant entre crochets
