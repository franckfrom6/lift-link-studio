# Test classification (Mode B)

Rules to classify each parsed test into one of five categories. The classification is stored in `docs/testing/test-registry.json` and drives downstream behavior (skip, run with caveat, require precondition).

## Categories

| Category | Meaning | Behavior |
|----------|---------|----------|
| `auto` | Pure Playwright, no special setup needed | Run normally |
| `auto-with-precond` | Auto but requires app state set up beforehand | Run after preconditions |
| `auto-ai-flaky` | Touches VOLT/AI — non-deterministic output | Run with relaxed assertions (presence, not exact content) |
| `auto-with-caveat` | Auto but imperfect simulation (backgrounding, force reload) | Run with `page.evaluate` simulation, flag in report |
| `manual-only` | Cannot be tested in Playwright at all | Skip with explanation, list in checklist |

## Classification rules

Apply rules in **order**. First match wins. Use case-insensitive matching unless specified.

### Rule 1 — manual-only (highest priority)

A test is `manual-only` if its text mentions any of:

- iOS silent mode / silent switch / ringer
- iOS PWA install / "add to home screen" / home screen
- Alarm volume / "clearly audible" / sound quality assessment
- Vibration / haptic feedback
- 3D touch / force touch
- Native iOS gestures (swipe-from-edge, etc.)
- Physical device requirement (camera roll access, photo picker, etc.)
- "On a real iPhone" / "on iOS" with physical-action context

Examples : T-A-36 (silent mode), T-A-37 (audible alarm), T-A-101 (iOS PWA install).

### Rule 2 — auto-ai-flaky

A test is `auto-ai-flaky` if its text mentions any of:

- VOLT / AI / LLM
- "Générer via" / "Generate via"
- Natural-language input that triggers a generation step
- Expected output is content generated by an LLM (where exact wording can't be asserted)

Examples : T-A-18 (VOLT creates session), T-A-56 (VOLT description → blocks).

For these tests, assertions must verify **presence and shape**, never **exact content**:
- ✅ "A new session appears on tomorrow's tile"
- ✅ "Blocks count > 0"
- ❌ "Block title equals 'Échauffement 15 min'"

### Rule 3 — auto-with-caveat

A test is `auto-with-caveat` if its text mentions any of:

- "Switch to another app" / "Background the app"
- "Force-reload" / "Force-refresh" / "Reload the page"
- "Press the Home button"
- "Wait N seconds then return" (with implied app eviction)
- Simulated iOS app lifecycle events

Examples : T-A-35 (rest timer background), T-A-43 (elapsed timer after app switch), T-A-44 (state restored after reload).

Implementation hints :
- "Force reload" → `await page.reload()`
- "Switch app" → `await page.evaluate(() => document.dispatchEvent(new Event('visibilitychange')))` + wait, then back
- "Wait then return" → cannot truly simulate eviction; flag this clearly in the report

### Rule 4 — auto-with-precond

A test is `auto-with-precond` if EITHER:

**(a) Its text mentions any of:**
- "An exercise you've done before" / previous session data
- "Programmed session" / "active program" / "assigned program"
- "A completed session"
- "Coach-defined" anything (macros, plan)
- "A day that has 2+ sessions"
- "A student with no program" / "no coach program assigned"
- "Linked coach" / "linked athletes"

**(b) Its parent section heading implies a contextual state (section-level precondition inheritance).** When a section's URL or theme implies that the user must already be in a specific app state, all tests in that section inherit the relevant precondition(s).

Section → inherited preconditions mapping:

| Section heading pattern | Inherited preconditions |
|------------------------|------------------------|
| "Live Session (`/student/session/:id`)" | `athlete-has-programmed-session-today` |
| "Session Preview (`/student/session/:id/preview`)" | `athlete-has-programmed-session-today` |
| "Weekly Calendar (`/student`)" | (none — empty state is valid) |
| "Progress (`/student/progress`)" | `athlete-has-completed-session` |
| "Nutrition (`/student/nutrition-plan`)" | `coach-defined-nutrition-plan` |
| "Student Detail (`/coach/students/:id`)" | `athlete-linked-to-coach` |
| "Program Editor" | `athlete-linked-to-coach` |
| "Programs Library (`/coach/programs`)" | (none) |

When a test has BOTH section-inherited preconditions AND its own (rule a) match, list all preconditions in the registry. Order doesn't matter for execution — all preconditions must be met before the test runs.

Examples : T-A-20 (preview requires programmed session), T-A-31 (previous weight suggestion), T-A-72 (nutrition plan from coach), T-A-32 (live session context — rule b).

Each `auto-with-precond` test must have its precondition mapped in `orchestration-coach-athlete.md`. If a new precondition pattern appears that's not yet mapped, flag it to the user instead of guessing.

### Rule 5 — auto (default)

If none of the above match, classify as `auto`. Pure Playwright, no special handling.

## Output format — test-registry.json

The registry is one JSON file at `docs/testing/test-registry.json`, structured as :

```json
{
  "generated_at": "2026-05-24T16:30:00Z",
  "scripts": {
    "coach-test-script.md": {
      "tests": {
        "T-C-01": {
          "category": "auto",
          "section": 0,
          "title": "Log in",
          "preconditions": [],
          "notes": ""
        },
        "T-C-39": {
          "category": "auto",
          "section": 3,
          "title": "Impersonation — Voir comme l'athlète (FIRST CLICK)",
          "preconditions": ["athlete-linked-to-coach"],
          "notes": "Known bug per script (FIRST CLICK fails). Test should expect failure on first click in current state."
        }
      }
    },
    "athlete-test-script.md": {
      "tests": {
        "T-A-30": {
          "category": "auto-with-precond",
          "section": 3,
          "title": "Session loads correctly",
          "preconditions": ["athlete-has-programmed-session-today"],
          "notes": ""
        }
      }
    }
  }
}
```

## Re-classification

When the user updates a script (e.g. adds a new test or changes wording), re-run classification on **only the changed/new entries**, not the whole registry. Preserve existing manual edits in `notes`.

When in doubt about a classification — ask the user. It's better to ask once than to silently miscategorize a test that gets run in the wrong category for months.

## Known-bug handling

Some tests explicitly document a "known bug" in their Watch-for section (e.g. T-C-39, T-C-40, T-C-42, T-A-39).

For these tests :
- Classify normally per the rules above
- In `notes`, mention "Known bug per script" with brief description
- In the generated spec, the test should still run — the value is to detect when the bug is fixed (test starts passing again), not to assume it'll fail forever
